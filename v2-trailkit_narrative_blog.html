TrailKit is a browser-based gear management application for outdoor recreation — built without a backend, without a framework, and without a development team.

This document tells that story. It is part process narrative, part design retrospective, and part artifact log. It traces every meaningful decision — from the first aesthetic prompt to the final version bump to v0.96 — and explains how the product evolved, what principles guided it, and what the experience of building software this way actually looks and feels like.

The app was built entirely in HTML, CSS, and vanilla JavaScript. No React. No build step. No npm install on the user's end. The entire application ships as a single .html file you can open in any browser.

Part 1

The Origin

A Screenshot and a Concept

The Founding Prompt

Every project has a first move. For TrailKit, it was a screenshot from a video game inventory system and a deliberately open-ended prompt:

help me create a css mockup of an 'inventory and loadout management tool' for outdoor recreation, based on the format in the attached screenshot from a video game. this will become a full webapp, but for a first step, I want to establish a clear look and feel. Then we will work on interactivity.

The context was specific: a Stash or Inventory panel on the left would hold a complete catalog of all outdoor recreation gear. A Loadout section would represent a specific packed configuration — the gear selected for a particular trip or activity. The prompt referenced backpacks, tents, sleeping bags, climbing gear, cycling equipment, and water sport accessories.

The instinct to separate visual design from functional development was deliberate and proved to be foundational. Before writing a single line of logic, the goal was to know what the thing should look like.

Version 0.01: The First Build

v0.01 — CSS mockup — aesthetic only, no interactivity

The first version established a design language that would persist through every subsequent iteration:

  • Dark tactical UI with Rajdhani as the display typeface and Share Tech Mono for numeric stats
  • Electric teal (#00c8b4) and amber (#f5a623) as the primary accent palette
  • A scanline overlay texture for a subtle screen aesthetic drawn from the game UI reference
  • A 3-column item grid in the Stash panel with 31 pre-populated sample items
  • A Pack Weight meter — a visual progress bar indicating total carried weight

Critically, the AI named the product. The name TrailKit emerged from the first response — not as a suggestion, but as the working title applied to the logo treatment in the UI header. It stuck immediately.

Design decision

Establish the name and visual identity in version 1. Both became fixed reference points for every subsequent decision.

Part 2

Visual Refinement

Versions 0.02 through 0.10

Iterating Toward a Visual Template

Versions 0.02 through 0.10 were almost entirely aesthetic. Each prompt was a precise list of changes: rename this, move that, change this color, remove this element. The process revealed something important — good UI iteration is granular. Each round of changes was small and surgical, never wholesale.

v0.02 — Controls and Layout
  • Added a prominent Add Item(s) button to the Stash header
  • Introduced Small / Medium / Large size indicators on item cards
  • Added a Configure Backpack button in the loadout area
  • Added Safety Items Packed and Medical Items Packed status indicators
  • Made the Activity Selector the dominant element in the top bar, with sport-specific accent colors
v0.03 — Identity and Navigation Cleanup

The MorganPCO / SYNCED profile chip in the upper right felt like scaffolding left over from a game UI. It was replaced with a MY TRAILKIT button styled in the signature teal (#00c8b4). The ACTIVE CONFIGURATION subtitle under LOADOUT was removed — the label alone was sufficient.

v0.04 — Naming and Controls

A cascade of naming and control decisions in a single prompt:

  • CURRENT LOADOUT → NEW LOADOUT
  • Added a Save New Loadout button in amber (#f5a623) next to the loadout dropdown
  • Added a + Activity button at the end of the activity strip
  • WATER activity replaced with CAMPING; SAVE LOADOUT → EXPORT TRAILKIT; STASH → YOUR GEAR
  • Removed numerical x6 quantity indicators from item cards
v0.05 — Dark / Light Mode

A single-line request with significant technical weight:

could we add a working Light Mode / Dark Mode toggle in the bottom bar?

This was the first functioning feature — a body.light CSS class toggle that overrides all custom properties at once, paired with a pill-style switch in the bottom bar. Two modes, one toggle, zero JavaScript frameworks.

v0.06–0.09 — Contrast, Slots, and Layout
  • Brightened text labels like HYDRATION BLADDER and FILTER SLOT for readability
  • Made empty slot outlines visible and consistent across the loadout
  • Standardized hover highlight behavior across all empty slots
  • Adjusted the Pack Weight meter labels: ULTRALIGHT → LIGHT, TRAIL → DAYPACK, EXPEDITION → HEAVY
  • Added a Max Weight stat to the backpack metadata
v0.10 — The Design Freeze

After ten iterations, a deliberate decision was made to lock the visual template:

this is a fantastic mock up — Version 10. I don't want to deviate from this — as features are added, I want them to be included in a way that leverages this UI, strictly adhering to the layout that has been developed since the initial prompt. Only add/remove/change visual elements from this Version 10 visual template when a feature or addition is requested that explicitly authorizes changes.

This was a pivotal moment in the project. Rather than letting feature development gradually erode the design, the visual system was declared complete. Future changes required explicit authorization. The phrase I am explicitly authorizing became a recurring pattern in subsequent prompts.

Part 3

Building Real Functionality

Versions 0.11 through 0.40

From Mockup to Application

Version 0.10 was a high-fidelity static mockup. Versions 0.11 onward turned it into an application. The transition required a complete rethinking of the codebase — not a patch on top of the mockup, but a ground-up rewrite of the JavaScript engine that powered the UI without disturbing the visual template.

The first functional build introduced:

  • Drag-and-drop between the inventory and loadout
  • Pack profiles with different slot layouts per backpack
  • Save and load named loadouts
  • Functional activity filtering
  • Pack weight calculation based on real item weight values

Slot System Logic

One of the most technically interesting design decisions was how items of different sizes are handled in the loadout compartments:

  • When an item occupying more than one slot is added to a compartment, the last N empty slots are hidden (not subsequent ones)
  • When the item is removed, those slots are restored
  • Compartment slots re-sort automatically — occupied slots condense toward the top, empty slots toward the bottom
  • The hydration bladder slot works differently — items larger than 1L hide proportional slots, items smaller add proportional slots back

This logic required careful sequencing. Add an item → hide tail slots → re-sort. Remove an item → restore tail slots → re-sort. The exception case (the bladder slot's volume-proportional behavior) was explicitly carved out.

The UI Restructure — Version 0.14

v0.14 — Major layout authorization — activity selector, side column, info tooltips

Version 0.14 was the largest single-prompt change in the project's history. It required explicit authorization and touched nearly every panel:

  • Removed the activity strip from the top bar entirely
  • Added an Activity dropdown between LOADOUT and the loadout selector
  • Further visually separated the YOUR GEAR column from the LOADOUT
  • Connected the right column (ESSENTIAL ITEMS, WORN / BODY, POCKET ITEMS) visually to the LOADOUT
  • Added an item info tooltip system for hover-over item details

The principle driving these changes: the three-column structure of the app should be visually self-evident. YOUR GEAR is a library. The LOADOUT panel and its right-column extensions are a workspace. Those two zones should feel distinct.

Feature Pruning — Removing What Doesn't Belong

it's time to strip away some functionality that isn't essential!

  • Removed Volume and Bladder stats from the backpack metadata
  • Changed FOOD + WATER back to just WATER
  • Removed Calories Packed, Electrolytes indicator, and Add Food button
  • Simplified bottle pockets to a single larger slot per pocket
  • Removed the CHEST / HIP BELT POCKETS section entirely

Pruning is as much a part of product development as adding. Each removed element was something that had seemed valuable at one point but ultimately added complexity without proportionate value.

The Data Model — Versions 0.23 and 0.24

Versions 0.23 and 0.24 defined the schema and rules that would underpin everything:

  • YOUR GEAR represents a user's total inventory across all activities
  • Items moved into a loadout become temporarily unavailable in YOUR GEAR — shown dimmed, not hidden
  • Items have exactly one Type: Item, Worn, Bottle, Bladder, Safety, Medical, Tools, or Backpack
  • Items can be Activity-Specific or universal (All Activities)

The file format was defined in XML — human-readable, self-documenting, and survives round-tripping through a text editor without surprises. A <trailkit> root containing <yourgear> and <loadouts>, with special item properties nested under a <special> tag.

Part 4

Edit Item and Add Item

Versions 0.50 through 0.70

The Edit Item Interface

The Edit Item feature was one of the most carefully specified interactions in the project. The design was precise: clicking Edit Item activates a hover mode where the cursor highlights any item with a bold yellow border and glow. Clicking an item opens a centered modal with full editing capabilities.

The modal displays and allows editing of:

  • Item name — editable text field
  • Item icon — large emoji display with an Edit button that opens a pure JavaScript emoji picker
  • Type — dropdown defaulting to Item, followed by Backpack, Bladder, Bottle, Safety, Medical, Tools, Worn
  • Activity — dropdown for All Activities or any defined activity
  • Slots, Weight, Description — numeric and text fields
  • Conditional special properties — shown only when item Type requires them

Type-conditional fields were a specific requirement — the modal should not show backpack configuration fields for a sleeping bag, or bottle capacity fields for a rain jacket. The UI adapts to the selected type in real time.

The Emoji Picker

Rather than relying on a third-party emoji library, the picker was built from scratch in vanilla JavaScript. The implementation uses named keyword-searchable emoji data covering 600+ emoji across all major Unicode categories. The search is live and filters results as you type.

This decision reflects a broader principle in TrailKit's construction: prefer self-contained solutions over external dependencies. The application has no npm dependencies, no CDN scripts, no external fonts beyond Google Fonts. Everything works offline.

Add Item — Mirroring Edit

Once the Edit Item modal was complete, the Add Item flow was specified to mirror it exactly — same fields, same layout, same behavior. The only differences:

  • Item Name and Description start empty
  • Type defaults to Item; Activity defaults to All Activities
  • Slots defaults to 1; Weight defaults to 0.1 kg
  • Bladder defaults to 2L; Bottle defaults to 1L

Using the Edit modal as a design reference for Add Item meant zero new design decisions — the interface was already proven. The Add and Edit flows are visually and behaviorally identical, differentiated only by their default state and save action.

Part 5

Sample Gear and State Persistence

Versions 0.63 through 0.75

The Sample Gear Toggle

Throughout development, TrailKit shipped with a set of pre-populated sample items — tents, sleeping bags, headlamps, trekking poles, and more — that made the app feel populated and demonstrated its capabilities. Rather than removing this sample data when the real user data model was introduced, it was preserved as a permanent, switchable dataset called SAMPLE GEAR.

  • A SAMPLE GEAR IN USE indicator and toggle button sit in the center of the top bar
  • When active, the left column switches from YOUR GEAR to SAMPLE GEAR
  • The toggle reads Switch to Your Gear when sample data is active, and Switch to Sample Gear otherwise
  • The sample dataset is baked into the application — always available regardless of a user's own inventory

This feature solves a real onboarding problem: a new user opening TrailKit for the first time sees an empty inventory and may not immediately understand the app's potential. Sample Gear provides immediate context.

Import and Export — The .trailkit File

State persistence was implemented through a custom XML file format with the .trailkit extension. The Import My TrailKit and Export My TrailKit buttons handle the full round-trip:

  • Export serializes the current YOUR GEAR inventory and all saved loadouts into the XML schema
  • Import parses a .trailkit file and restores the full state
  • The schema is human-readable and can be edited in a text editor

The Export My TrailKit button opens a centered modal with three options:

  • Download My TrailKit — saves the .trailkit XML file
  • Export as Packing Lists — generates a standalone HTML file with checkbox-based packing lists for each loadout
  • Export as CSV — exports the inventory as a spreadsheet-compatible file

The Packing List Export

The packing list export deserves special mention. It generates a complete, standalone HTML file — not a template, but a fully functional document with:

  • A separate section per saved loadout
  • Checkboxes for each item
  • A light/dark mode toggle
  • A Print Mode that switches to light mode and strips all colors and borders for clean printing

The packing list file is itself a polished product — not a raw data dump. It could be saved to a phone, printed before a trip, or shared with a hiking partner.

The decision to formally name a version 0.95 marked a shift in the project's tone. The application was functionally complete in its core scope. What remained were refinements:

  • A version indicator added to the bottom bar

Version 0.96 completed two features and added one:

  • A Print Mode was added to the packing list — activates light mode and removes all styling for a minimalist printed output

These were genuinely finishing touches — the kind of polish that distinguishes a complete product from a functional prototype.

Part 7

Reflections on the Process

What Made This Work

Several practices made this collaboration unusually productive:

  1. Design before logic

    Ten versions of pure aesthetic iteration before a single line of business logic. This seems slow but is actually fast — changing a color is trivial; changing a data model is not. Getting the design right first meant the subsequent logic could be built to serve a known, stable UI rather than the UI being retrofitted around logic.

  2. Explicit authorization for UI changes

    The phrase I am explicitly authorizing became a shared protocol. After Version 0.10 locked the visual template, any prompt that changed the UI had to say so clearly. This prevented feature creep from eroding the design and forced deliberate decisions about what a new feature should look like before it was built.

  3. Pruning as a first-class activity

    The willingness to remove features — Calories Packed, Electrolytes, Food section, Chest/Hip Belt Pockets — kept the application focused. Adding features is easy and feels good. Removing them requires discipline. The pruning pass at Version 0.21 made the final product better.

  4. Small, precise prompts

    The most effective prompts in this project were lists of specific changes, not open-ended requests. Remove this. Rename that. Change this color to #f5a623. This precision minimized misinterpretation and kept iterations tight.

  5. The single-file constraint

    Choosing to build the entire application as a single HTML file was a constraint that simplified everything. No build step. No deployment. No dependencies. The constraint made the project tractable for a non-developer and produced a genuinely portable artifact.

Closing

TrailKit went from a screenshot reference and a single sentence to a fully functional, visually polished, self-contained web application in 23 iterations. It has a custom file format, an emoji picker, drag-and-drop inventory management, multi-loadout support, light and dark modes, a standalone packing list exporter with its own print mode, and a versioned release history.

None of that required a development team, a framework, a deployment pipeline, or anything beyond a browser and a conversation.

The tool that built TrailKit is the same kind of tool TrailKit is designed to support: something that removes the friction between having an idea and making it real.
